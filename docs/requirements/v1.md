# Iteration Requirements — V1

## Iteration: v1 — Register, connect, chat

- **Status:** Active
- **Goal:** A user can receive an invite, register, add friends, and exchange
  real-time text messages on a server-hosted app behind TLS.

---

## In scope

- **Walking skeleton** — trivial FE + BE deployed behind nginx/TLS with GitHub
  Actions CI wired up, before any feature work starts.
- **Invite-gated registration** — a user can only sign up with a valid invite
  code. An admin generates codes via a script or CLI command.
- **Auth** — email + password sign-up and login, JWT access token + refresh
  token rotation.
- **Username** — each user picks a unique `@username` (display handle).
  Login identity (email) is separate from display identity (username).
- **Friend requests** — mutual, with explicit accept/reject. Users cannot
  message each other until both sides are connected.
- **Real-time 1:1 text messaging** — messages sent and received over
  WebSocket.
- **Message persistence** — all messages stored in PostgreSQL and visible
  after a page refresh or next login (no history is lost).

---

## Explicitly out of scope

- End-to-end encryption (V1 is TLS-only)
- Media (images, video)
- Group chats
- Presence indicators, typing indicators, read receipts, reactions
- Password reset / forgot-password flow
- Profile photos or bios
- Invite expiry, multi-use invite codes, or an invite management UI
- Microservices, message queues, horizontal scaling

---

## Acceptance criteria

- [ ] An admin can generate an invite code (via script or CLI).
- [ ] A user can register with a valid invite code (email + password +
      username). Registration with no code or an already-used code is rejected.
- [ ] A user can log in and receive a JWT access token + refresh token.
- [ ] Token refresh works; an expired access token can be exchanged for a new
      one using the refresh token.
- [ ] A user can search for another user by `@username` and send a friend
      request.
- [ ] The recipient can accept or reject the request.
- [ ] Two mutually connected users can exchange text messages in real time.
- [ ] Messages are persisted: refreshing the page or logging out and back in
      shows the full history.
- [ ] The app is accessible over HTTPS/WSS from the server (TLS terminated
      at nginx).
- [ ] GitHub Actions CI runs lint + type-check + tests on every PR and blocks
      merge on failure.

---

## Open decisions

- [x] **JWT storage** — httpOnly cookie. Refresh token is set by the server as
      a `Set-Cookie` header; JavaScript never touches it. See ADR-0002.
- [x] **WebSocket auth** — short-lived ticket. Client calls a REST endpoint
      to get a one-time token (~30s TTL), then sends it in the WS upgrade
      request. HTTP auth stays in HTTP; WS just validates the ticket.
      See ADR-0003.
- [x] **Invite code distribution** — admin CLI script
      (`uv run python scripts/create_invite.py`). No extra feature code.

---

## Notes / risks

- **Walking skeleton is PR #1.** No feature work until FE + BE + CI + TLS are
  all live end-to-end. This de-risks deployment surprises early.
- **Invite codes add scope.** Keep the implementation minimal: a simple
  `invite_codes` table, single-use, no expiry in V1. Resist expanding this.
- **WebSocket auth is easy to get wrong.** Resolve the ADR before writing any
  WS code — it affects the backend auth middleware design.
- **Message pagination is not in scope** but the schema should not make it
  impossible (use a `created_at` timestamp and an auto-increment or UUID PK).
